Executive Summary

The Portfolio Construction App is a full-stack platform for managing investment portfolios using public ETFs and stocks. It connects to real brokerage accounts via SnapTrade, syncs positions and activities, enables portfolio grouping with target allocations, calculates drift, automates rebalancing, and provides rich analytics through a customizable dashboard.

Who Is This For?

Individual Investors
Connect brokerage accounts, view consolidated positions, track performance across accounts, and manage portfolio allocations with automated rebalancing.
Portfolio Managers
Build model portfolios, apply them to client accounts, monitor drift from targets, execute batch trades, and generate performance reports.

Core Value Proposition

  • Multi-Account Consolidation — Connect multiple brokerage accounts, see aggregated positions and true exposure
  • ETF Look-Through — Decompose ETFs into underlying stock holdings for real sector and geographic exposure analysis
  • Automated Rebalancing — Set target allocations, detect drift, generate and execute trade orders to maintain targets
  • Rich Analytics — Customizable dashboard with 23 widgets, performance tracking (TWR), risk metrics, benchmark comparison
  • Data Enrichment — Automated ingestion from EODHD, Alpha Vantage, and ETF.com for comprehensive instrument data

Authentication & Security

JWT-based stateless authentication with refresh tokens, OAuth2 (Google), and comprehensive security measures.

Authentication Flow ▶ 
sequenceDiagram
  participant U as User
  participant F as Frontend
  participant A as AuthController
  participant AS as AuthService
  participant DB as Database

  U->>F: Enter email + password
  F->>A: POST /auth/login
  A->>AS: login(credentials)
  AS->>DB: Find user by email
  AS->>AS: Verify Argon2id hash
  AS->>AS: Check lockout status
  AS->>DB: Create refresh token
  AS->>AS: Generate JWT (HS512, 60min)
  A-->>F: Set-Cookie: access_token, refresh_token (HttpOnly)
  F->>F: Store user in Zustand authStore
  Note over F: Auto-refresh before expiry via useSessionManager

Data Sync Cycle

  1. Connection established — SnapTrade OAuth flow creates broker authorization
  2. Position fetch — Manual trigger or nightly scheduler fetches current positions via PositionFetchService
  3. Activity ingestion — ActivityIngestionService pulls trades, dividends, fees since last sync
  4. Balance snapshot — Daily balance snapshot stored in broker_balance_snapshots
  5. Drift check — If portfolio group has targets, DriftCalculationService calculates accuracy

Key Concepts

ConceptDescription
Portfolio GroupA named collection of linked brokerage accounts treated as a single portfolio
Target AllocationDesired percentage for each symbol (e.g., VTI: 40%, VXUS: 30%, BND: 30%)
Model PortfolioReusable template of allocations (system-defined or user-created) that can be applied to groups
AccuracyPercentage match between actual holdings and target allocations (100% = perfect match)
DriftDeviation from targets. Triggers rebalancing when accuracy drops below threshold
Rebalance EventRecord of a rebalancing action (trigger type, accuracy before/after, status)
Excluded AssetsSymbols to ignore in drift calculations (e.g., cash-like positions)
Model Portfolios ▶

Model portfolios are reusable allocation templates with risk levels (LOW, MODERATE, HIGH, EXTRA_HIGH). They can be:

  • System models — Pre-defined by the platform (e.g., "Conservative Growth", "Aggressive Tech")
  • User models — Custom models created by individual users
  • Applied to accounts — A model can be linked to broker connections, which then tracks drift against the model's allocations

Trade Execution Flow

  1. Rebalance calculation — RebalanceService compares current positions to targets, generates trade recommendations
  2. Order creation — OrderExecutionService creates TradeOrder records with batch UUID
  3. Broker submission — Orders sent to SnapTrade via placeOrder() adapter method
  4. Status tracking — Frontend polls GET /api/v1/trading/groups/{id}/orders every 10-15 seconds
  5. Completion — Filled orders update position data on next sync

Resilience Features

  • Rate limiting — Guava rate limiter for EODHD; custom token bucket for Alpha Vantage
  • Retry with backoff — Exponential backoff (1s to 300s) with jitter for transient failures
  • Circuit breakers — Resilience4j circuit breakers prevent cascade failures
  • Hash-based dedup — Redis-cached content hashes skip unchanged records
  • Batch processing — Configurable batch sizes with inter-batch delays
  • Error tracking — Failed items recorded in ingestion_errors with error type and symbol
